# Memoria-Agent 核心优化审计（第四轮）

## 本轮目标

继续对照 Akashic 的 `memory2/dedup_decider.py`、`memory2/memorizer.py`、`memory2/post_response_worker.py` 和 `memory2/store.py`，解决 Memoria 把所有高相似记忆都当作重复、无法表达用户信息变化的问题。

## 已接入能力

### 记忆生命周期

`memories` 新增 `status`、`reinforcement`、`supersedes_id` 和 `last_reinforced_at`。所有历史数据库启动时自动迁移，正常召回只读取 `active`，被替代条目标记为 `superseded` 并保留。

### 两阶段一致性判定

第一阶段使用规范文本、Jaccard 和 embedding 余弦相似度筛选最多三个同类型候选；第二阶段使用 fast 模型输出严格 JSON 决策：

| 决策 | 行为 | 安全约束 |
|---|---|---|
| `create` | 创建独立条目 | 默认和失败回退策略 |
| `reinforce` | 强化已有条目 | 目标必须来自候选且相似度至少 0.78 |
| `supersede` | 创建新版本并退休旧版本 | 仅 preference/profile/goal/procedure，相似度至少 0.55 |

模型输出的目标 ID 必须在预筛候选集合中。非法动作、越界 ID、解析失败或供应商失败全部降级为 `create`，避免误删用户记忆。

### 原子替代与审计

新版本写入、旧版本状态更新和 `memory_replacements` 审计记录在一个 `BEGIN IMMEDIATE` 事务内完成。审计表保留新旧正文快照、关系、判定原因和时间，即使之后删除某个条目仍可解释发生过什么。

### 召回热度

RRF 融合排序加入有上限的 reinforcement 小幅加权。同类信息被多次确认后排名略微提升，但不会压过明显更相关的关键词或向量结果。

## API 与前端

- API 版本提升到 `0.4.0`。
- `GET /api/memories` 支持 `status=active|superseded|all`。
- `GET /api/memories/{id}/history` 返回替代历史。
- `POST /api/memories` 明确返回 `created/reinforced/superseded`，重复强化不再伪装成 HTTP 错误。
- Memory 响应公开状态与强化次数，但不公开 embedding。
- 前端记忆卡显示强化次数，运行追踪页分别显示有效记忆和已替代记忆数量。

## 验证

- 精确重复能够强化且不新增行。
- fast 模型判定替代后，旧条目退出默认查询，新条目指向旧条目。
- 替代历史包含新旧正文和原因。
- 旧版 SQLite schema 可以原地迁移。
- 使用当前真实 fast 模型验证“由咖啡改为红茶”得到 `supersede`。

## 下一步建议

下一项优先做流式响应与取消令牌，因为当前浏览器断开后上游模型和工具仍会继续执行。随后建立中文记忆评估集，再决定是否值得接入 HyDE、query rewrite 和更复杂的 consolidation。
