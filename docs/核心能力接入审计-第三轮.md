# Memoria-Agent 核心能力接入审计（第三轮）

> 这是第三轮完成时的历史快照；sqlite-vec 等后续实现以第六轮生产化文档为准。

## 1. 本轮结论

本轮重新检查了 `akashic-agent/agent/retrieval`、`memory2/embedder.py`、`memory2/retriever.py`、`memory2/store.py`、`memory2/dedup_decider.py`、工具循环和中断相关代码。排除飞书、Telegram、QQ、主动推送和 Peer Agent 后，Memoria 最明显的真实缺口是：配置中已有 embedding 模型，但核心链路从未调用它。

这项缺口已经落地，不再只是规划：Memoria 现在具备向量持久化、历史回填、关键词/向量双路召回、RRF 融合、语义去重和故障降级，并已用当前 `config.toml` 的 `text-embedding-v3` 做过真实调用验证。

## 3. 下一批值得接入的核心能力

### P0：记忆版本、强化与 supersede

Akashic 的 post-response worker 不只判断重复，还能强化相同事实、合并相近信息、把过时规则标记为 superseded。Memoria 当前只有单条 CRUD，如果“喜欢简洁回答”后来变成“希望详细解释”，高相似度去重可能掩盖冲突。

建议先扩展 schema：`status`、`supersedes_id`、`reinforcement`、`last_reinforced_at`、`valid_from`。随后让模型只对高相似候选执行结构化 `ADD/MERGE/SUPERSEDE/NOOP` 决策，并保留人工撤销路径。这是下一轮最高优先级。

### P1：流式响应、取消令牌和断连停止

Akashic 的 looping/interrupt 会处理中断。Memoria 当前 `/chat` 等待完整回复；浏览器断开后，上游模型与工具仍可能继续运行。建议增加 SSE 流式端点、turn cancel token、客户端断连检测，并保证中断时 trace 状态为 `cancelled`。

### P1：记忆评估集

Akashic 仓库包含 LongMemEval 和 PersonaMem 评估入口。Memoria 应先建立一个小型中文数据集，分别统计关键词召回、向量召回和融合召回的 Recall@K、MRR、错误记忆注入率，再调整 0.15、0.86、0.94 等阈值。没有评估时不应继续堆叠 HyDE 和 query rewrite。

### P2：后台记忆任务

当前主回复保存后仍同步等待快速模型提取和 embedding。建议引入进程内可靠队列或持久任务表，把提取、向量化、合并和 consolidation 移出请求关键路径，并增加失败重试与幂等键。

### P2：procedure 识别与注入策略

Akashic 对 procedure、profile、episode 使用不同规则。Memoria 已有 `procedure` 类型，但还没有专门提取和“何时注入”的策略。应在真实数据证明有需要后，增加按类型限额和注入位置，而不是让所有类型只按统一排名进入 prompt。

## 4. 当前不建议接入

| 能力 | 原因 |
|---|---|
| 飞书、Telegram、QQ 等 Channel | 用户明确要求先完善本地核心 |
| 主动推送、Drift、Scheduler | 会引入行为边界和外发风险，不是对话正确性的前提 |
| Peer Agent / Subagent Manager | 当前单 Agent 工具闭环足够，过早引入会扩大并发状态面 |
| 完整插件发现和依赖拓扑 | 现有 Pipeline 已满足内部扩展，尚无第三方插件生态 |
| Shell、任意文件系统工具 | Dashboard 没有成熟权限沙箱，风险大于收益 |
| HyDE、多轮 query rewrite | 增加模型成本和延迟；先以评估证明基础双路召回不足 |
