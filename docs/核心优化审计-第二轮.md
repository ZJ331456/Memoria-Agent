# Memoria-Agent 核心优化审计（第二轮）

## 结论

本轮再次对照 `akashic-agent` 的 Provider、PromptAssembler、PassiveTurn、tool runtime、tool loop guard 和 post-response memory worker。Memoria 已经具备生命周期、工具、记忆和 trace 基础，本轮最值得迁移的是故障保护，而不是继续复制大型模块树。

## 已落地

### Provider 可靠性

新增最多两次指数退避，只重试超时、连接异常、429 和 5xx。401、400 等确定性错误立即失败。上下文过长与内容安全分别抛出 `ContextLengthError`、`ContentSafetyError`，不再全部包装成模糊的 HTTP 错误。

### 上下文预算

每次模型调用前限制 prompt 字符数，保留 system 和最近消息，截断异常大的工具结果。裁剪会同时检查 assistant tool call 和 tool result 是否完整，避免产生供应商无法接受的悬空协议消息。上下文超长时使用 45% 紧急预算再尝试一次。

### 工具循环保护

以“工具名 + 排序后的 JSON 参数”作为稳定签名。相同批次允许执行两次，第三次不再触发副作用；Runtime 会补齐 tool result，并调用模型输出阶段性结论。

### Trace 脱敏

Trace 不再复制召回记忆正文，只保留 ID、类型、重要度和来源。工具参数中的 key、token、password、secret、authorization 等字段递归替换，字符串中的 Bearer 和 `sk-` 凭据也会遮蔽。

## 暂不照搬

### 完整插件系统

Memoria 的 `Pipeline` 已提供核心扩展点。当前没有第三方插件生态，直接复制 discovery、回滚、manifest 和依赖拓扑会显著增加维护面，收益较低。

### Post-response supersede

Akashic 的 worker 能识别“旧规则错了”并 supersede 记忆，但它依赖记忆状态、reinforcement、向量阈值和 protected IDs。Memoria 当前记忆表尚无版本状态，直接实现可能错误删除，因此应先完成记忆版本模型再迁移。

### 主动推送和 Drift

它们是应用行为而非对话核心稳定性。本项目当前目标是本地核心功能验证，继续保持不启用。

### 完整 token tokenizer

不同供应商 tokenizer 不统一。当前字符预算可预测且零依赖；未来需要精确成本统计时，再按模型适配 tokenizer。

## 下一轮建议

优先级依次是：记忆版本与 supersede、真实 embedding 混合检索、流式响应与取消令牌、后台记忆 consolidation。每项都应保持现有 `/chat` API 兼容。
