# Memoria-Agent 项目规划说明书

## 1. 文档定位

本文档不是对 `akashic-agent` 的简单抄写，而是基于其真实实现抽象出一份可用于 `Memoria-Agent` 从 0 开始落地的工程规划文档。

目标有三个：

- 作为 `Memoria-Agent` 的架构蓝图，明确模块边界、核心对象和启动链路。
- 作为实现顺序指南，帮助后续按阶段完成 MVP、Beta 和增强版功能。
- 作为对标参考，保留 `akashic-agent` 中已经被验证过的设计经验，避免重复踩坑。

一句话定位：

`Memoria-Agent` 是一个以“长期记忆 + 主动触达 + 插件化生命周期 + 可观测工作台”为核心的个人 AI Agent Runtime。

## 2. 参考来源与设计原则

### 2.1 主要参考来源

本规划主要参考 `\root\project_job\akashic-agent` 的以下真实实现方向：

- `main.py`：入口模式拆分、默认启动行为、`setup/init/dashboard/cli/gateway`。
- `bootstrap/app.py`：`AppRuntime` 应用级启动和关闭编排。
- `bootstrap/tools.py`：`CoreRuntime` 装配、依赖收束、工具集接入。
- `agent/looping/core.py`：被动对话主循环与中断态。
- `proactive_v2/loop.py`：主动推送主循环与 `PROACTIVE_CONTEXT.md`。
- `agent/plugins/manager.py`：插件发现、加载、回滚、工具与生命周期注入。
- `_handbook/plugins-tutorial.md`：PhaseModule、EventBus、tool hook 设计。
- `_handbook/memory-markdown.md`：Markdown 记忆文件层与 consolidation 机制。

### 2.2 迁移原则

`Memoria-Agent` 在设计时应遵循以下原则：

- 优先复用抽象，不盲目复刻目录和文件名。
- 先搭建运行时骨架，再补强插件、主动推送和多通道。
- 先保证单用户可用，再考虑多用户、多租户和多 Agent 协作。
- 所有高副作用能力都必须可审计、可限制、可关闭。
- 所有长期记忆都要同时满足“人类可读”和“程序可检索”。

### 2.3 非目标

以下内容不建议在第一阶段作为必须项：

- 多租户 SaaS 化部署。
- 完整插件市场和热更新体系。
- 复杂图谱记忆和跨 Agent 共享记忆。
- 大而全的通道支持。
- 复杂权限沙箱。

## 3. 项目概述

`Memoria-Agent` 面向个人长期陪伴、任务协作和主动提醒场景。它不只是一个问答机器人，而是一套长期运行的 Agent 系统，包含：

- 被动对话链路：收到消息后完成上下文准备、记忆检索、工具调用、回复生成和会话持久化。
- 主动推送链路：在用户未发消息时根据规则、上下文和数据源判断是否要主动触达。
- 长期记忆系统：同时保留 Markdown 文件层与语义向量层。
- 插件扩展系统：允许外部模块插入生命周期、工具、通道和 Dashboard 面板。
- 观察与运维系统：通过 Dashboard、日志、事件流和数据库记录运行状态。

项目核心目标：

- 让用户通过 Telegram、CLI、IPC 等渠道与 Agent 对话。
- 让 Agent 在单轮对话中具备记忆检索、工具执行、流式回复和中断恢复能力。
- 让 Agent 在空闲时能执行主动推送或 Drift 类后台任务。
- 让长期记忆可读、可查、可修正、可优化。
- 让整个系统便于二次开发、调试和持续演进。

## 4. 产品边界与阶段目标

### 4.1 MVP 范围

第一版 `Memoria-Agent` 建议只做以下闭环：

- 单用户。
- 单主通道，优先 Telegram 或 CLI。
- 被动对话主链路可用。
- Markdown 记忆可写可读。
- 向量记忆支持基础检索。
- 少量高价值工具可用，例如文件、网页读取、记忆工具、提醒工具。
- Dashboard 支持查看会话、消息和记忆。

### 4.2 Beta 范围

在 MVP 跑通之后，第二阶段补强：

- 主动推送闭环。
- `PROACTIVE_CONTEXT.md` 规则面板。
- 插件机制。
- 观察日志和错误追踪。
- Memory Optimizer。
- 更多通道和插件工具。

### 4.3 长期目标

- 多通道身份合并。
- 多 Agent 协作。
- 更成熟的权限和审计机制。
- 更强的记忆评估、回放和调优能力。
- 插件生态和 Dashboard 工作台化。

## 5. 技术栈建议

| 层级 | 技术与组件 | 作用 |
|---|---|---|
| 运行语言 | Python 3.12 | Agent Runtime、记忆、插件、通道、主动推送 |
| 异步运行 | asyncio | 消息循环、后台任务、通道协作 |
| 后端接口 | FastAPI、Uvicorn | Dashboard API、静态资源服务 |
| LLM 接入 | OpenAI-compatible API | 主模型、轻量模型、视觉模型、向量模型 |
| 调度 | APScheduler 或自研 asyncio loop | 定时任务、提醒、优化器 |
| 数据存储 | SQLite | 会话、消息、主动推送状态、事件日志 |
| 语义检索 | sqlite-vec 或兼容向量方案 | 向量记忆存储与召回 |
| 可读记忆 | Markdown 文件 | 长期记忆、近期上下文、主动规则 |
| 前端 | React + Vite + TypeScript + Tailwind | Dashboard 页面与插件面板 |
| 测试 | pytest、pyright、eslint、tsc | 单测、类型检查、前端质量控制 |

技术选型原则：

- 优先单机友好和低运维成本。
- 优先可替换接口，不把实现写死在业务流程里。
- 优先选择对 Agent 运行时友好的异步模型。

## 7. 运行模式规划

`main.py` 建议作为统一入口，解析 `--config`、`--workspace`、`--host`、`--port` 等参数，然后根据命令切换模式。

| 命令 | 行为 |
|---|---|
| `python main.py setup` | 运行交互式配置向导 |
| `python main.py init` | 初始化工作区和默认配置 |
| `python main.py gateway` | 启动完整 Agent 服务 |
| `python main.py dashboard` | 单独启动 Dashboard |
| `python main.py cli` | 连接运行中的 Agent IPC socket |
| `python main.py --inspect-modules` | 打印生命周期模块依赖树 |
| `python main.py` | 默认启动完整服务 |

工作区建议：

- 默认路径：`~/.memoria/workspace/`
- 支持 `--workspace` 指定自定义路径
- 所有运行时数据、记忆、数据库、附件和 trace 文件都落在 workspace 下

## 8. 启动装配总流程

建议直接沿用 `akashic-agent` 的“双层运行时”思路：

- `AppRuntime` 负责应用级聚合与关闭。
- `CoreRuntime` 负责核心能力装配。

推荐启动流程：

```text
main.py
  → Config.load(config.toml)
  → build_app_runtime(config, workspace)
  → AppRuntime.start()
      → configure shared HTTP resources
      → build_core_runtime()
          → MessageBus / EventBus
          → LLM providers
          → SessionManager / PresenceStore
          → MemoryRuntime
          → ToolRegistry
          → AgentLoop
          → PluginManager
      → core.start()
          → MCP connect background
          → peer agent discovery
          → plugin load_all
          → inject phase modules / tool hooks
      → start_channels()
      → start background tasks
          → AgentLoop.run
          → MessageBus.dispatch_outbound
          → Scheduler.run
          → MemoryOptimizer
          → Dashboard server
          → ProactiveLoop
```

### 8.1 核心运行时对象

| 对象 | 角色 |
|---|---|
| `AppRuntime` | 应用级生命周期管理器，统一启动和关闭 |
| `CoreRuntime` | 核心依赖容器，负责组装核心服务 |
| `MessageBus` | 通道与 AgentLoop 之间的消息总线 |
| `EventBus` | 生命周期事件、工具事件、观测事件总线 |
| `AgentLoop` | 被动对话主循环 |
| `ProactiveLoop` | 主动推送主循环 |
| `SessionManager` | 会话与消息持久化 |
| `MemoryRuntime` | 记忆系统统一入口 |
| `ToolRegistry` | 工具注册和查询中心 |
| `PluginManager` | 插件发现、加载、回滚和注入 |
| `ChannelHost` | 各通道统一管理器 |

### 8.2 为什么必须分成 AppRuntime 和 CoreRuntime

这样做有四个好处：

- 核心逻辑能在测试中单独启动，不依赖所有通道和 Dashboard。
- 运行模式拆分更自然，`dashboard`、`cli`、`inspect-modules` 更易实现。
- 后续把部分能力拆成独立进程或远程服务会更容易。
- 启动失败和关闭顺序更清晰，不容易出现资源泄露。

## 9. 被动对话链路规划

被动对话是用户主动发消息后的主路径。设计上建议保持 `MessageBus -> AgentLoop -> Pipeline -> Outbound` 这条链路。

### 9.1 端到端流程

```text
用户消息
  → Channel Adapter
  → MessageBus.publish_inbound
  → AgentLoop.consume_inbound
  → 获取或创建 Session
  → 建立 TurnInterruptState
  → PassiveTurnPipeline.run
      → BeforeTurn
      → BeforeReasoning
      → PromptRender
      → Reasoner
          → LLM 调用
          → tool call loop
              → BeforeStep
              → ToolHook
              → Tool.execute
              → AfterStep
      → AfterReasoning
      → AfterTurn
  → MessageBus.publish_outbound
  → MessageBus.dispatch_outbound
  → Channel 发送给用户
```

### 9.2 生命周期 Phase

建议保留 7 个可插拔 Phase：

| Phase | 核心职责 | 插件可做的事 |
|---|---|---|
| `BeforeTurn` | 获取会话、准备上下文、处理命令、中断和 abort | 命令拦截、写 hint、直接回复 |
| `BeforeReasoning` | 同步工具、检索记忆、补充 extra hints | 改可用工具、追加规则、拒绝进 LLM |
| `PromptRender` | 组装 system prompt 和上下文框架 | 注入段落、协议、角色规则 |
| `BeforeStep` | 每一轮 LLM/tool step 前检查 | 提前停止、注入 step hint |
| `AfterStep` | 每一轮工具执行后做 telemetry | 记录行为、调整策略 |
| `AfterReasoning` | 处理最终回复、持久化消息、构造出站内容 | 清理回复、追加媒体、写 extra |
| `AfterTurn` | fanout committed 事件、收尾 | 审计、触发后台工作、写 turn extra |

### 9.3 Reasoner 与工具循环

`Memoria-Agent` 的 `Reasoner` 需要具备以下能力：

- 渲染最终上下文。
- 读取当前轮可用工具 schema。
- 调用主模型并解析工具调用。
- 把工具结果写回消息列表。
- 循环直到产出最终回复或达到迭代上限。
- 支持流式输出。

建议保留以下约束：

- `max_iterations` 可配置。
- `tool_search_enabled` 可配置。
- 工具执行前统一经过 hook 链。
- 工具批量调用也要保留上下文信息，便于风控和观测。

### 9.4 中断机制

建议从第一阶段就支持中断：

- `AgentLoop` 维护每个 `session_key` 的 active task。
- 用户发送 `/stop` 或通道级中断命令时取消当前任务。
- 保存 partial reply 和 thinking 状态。
- 用户后续继续发消息时可以基于中断态接续处理。

这个能力对长上下文、多工具、多步骤任务非常有价值。

## 10. 主动推送链路规划

主动推送建议作为独立 loop，而不是塞进被动消息循环里。

### 10.1 设计目标

主动推送要解决的问题不是“定时发消息”，而是“什么时候值得打扰用户”。

因此建议把主动链路拆成：

- 调度：何时 tick。
- 数据获取：拿什么候选内容。
- 判断：是否发送、发什么。
- 去重与确认：避免重复、噪声和刷屏。
- 投递：通过统一消息出口发送。

### 10.2 调度与状态

建议保留以下机制：

- 用户刚互动后降低主动频率。
- 用户长时间未互动后提高主动频率。
- tick 引入 jitter，避免机械固定间隔。
- 独立的 `proactive.db` 记录 tick、delivery 和 trace。
- `PROACTIVE_CONTEXT.md` 作为主动规则文件，每轮读取。

### 10.3 推荐处理流程

```text
ProactiveLoop tick
  → ProactiveKernel.run_tick
  → ProactiveTurnPipeline.run
      → Gate
      → Fetch
      → Judge
      → Resolve
      → Deliver
```

每个阶段职责建议如下：

| 阶段 | 职责 |
|---|---|
| `Gate` | 判断是否应该进入本轮主动处理 |
| `Fetch` | 获取 alert/content/context 三类候选数据 |
| `Judge` | 用规则和 LLM 判断是否值得发消息 |
| `Resolve` | 做去重、ACK、skip/reply 决策 |
| `Deliver` | 通过统一消息出口投递 |

### 10.4 三类数据源

| 数据类型 | 语义 | 处理方式 |
|---|---|---|
| `alert` | 高优先级告警 | 更强倾向发送，发送后 ACK |
| `content` | 信息流候选 | 逐条判断、分类、过滤、草拟 |
| `context` | 辅助上下文 | 用于判断，不等同于必须推送 |

### 10.5 Drift 空闲任务

当没有可推送内容时，建议支持 Drift：

- 审计或整理记忆。
- 更新用户画像。
- 执行 `skills/` 下的 `SKILL.md`。
- 做自检和规则优化。
- 生成需要后续主动触达的候选提醒。

Drift 应受同一套工具 hook 和审计机制约束，避免后台任务失控。

## 11. 记忆系统规划

建议保留“双层记忆”架构：

- Markdown 文件层：给人看，给 system prompt 注入稳定上下文。
- 向量检索层：给机器搜，支持语义召回、去重和后台抽取。

### 11.1 Markdown 记忆文件

建议保留这些核心文件：

| 文件 | 作用 |
|---|---|
| `MEMORY.md` | 长期稳定记忆，低频更新，直接参与 system prompt |
| `SELF.md` | Agent 自身设定、行为边界、关系定义 |
| `HISTORY.md` | 按时间线记录关键事件 |
| `PENDING.md` | 待归档事实缓冲区 |
| `RECENT_CONTEXT.md` | 近期上下文摘要 |
| `PROACTIVE_CONTEXT.md` | 主动推送规则和偏好 |

### 11.2 Consolidation 流程

建议沿用 `akashic-agent` 的思路：

```text
TurnCommitted
  → 选择需要归档的消息窗口
  → LLM 提取 history_entries / pending_items
  → 写 HISTORY.md / PENDING.md / RECENT_CONTEXT.md
  → 更新 session.last_consolidated
  → 发出 ConsolidationCommitted
```

关键设计理由：

- `PENDING.md` 作为缓冲区，避免 `MEMORY.md` 高频更新破坏 prompt cache。
- `RECENT_CONTEXT.md` 用来保存近期讨论主线，降低模型重复阅读原始历史的成本。
- `HISTORY.md` 提供可追溯时间线。

### 11.3 Memory Optimizer

建议把 `PENDING.md -> MEMORY.md` 归档做成独立后台任务：

- 定时执行。
- 读取 `MEMORY.md + PENDING.md`。
- 做去重、合并、更正和重写。
- 更新 `MEMORY.md` 后清空 `PENDING.md`。

这样能在“记忆完整度”和“prompt cache 稳定性”之间取得平衡。

### 11.4 向量记忆层

建议最少包括这些组件：

| 组件 | 职责 |
|---|---|
| `MemoryStore2` | SQLite + 向量索引存储 |
| `Embedder` | 调 embedding 模型生成向量 |
| `Retriever` | 做检索、过滤、证据组织 |
| `QueryRewriter` | 把用户问题改写成更易检索的问题 |
| `Memorizer` | 写入、去重、替换、强化 |
| `PostResponseMemoryWorker` | 对话后抽取新记忆 |

### 11.5 记忆接口设计

建议把记忆能力抽象成 4 类 API：

- `MemoryIngestApi`
- `MemoryRetrievalApi`
- `MemoryWriteApi`
- `MemoryAdminApi`

这样 Dashboard、Tool、Pipeline、Plugin 都只依赖协议，不绑定具体实现。

## 12. 插件系统规划

插件是 `Memoria-Agent` 的关键扩展机制。即使第一版不完整实现，接口也要先设计好。

### 12.1 插件加载流程

```text
PluginManager.discover
  → 扫描 plugins/ 下包含 plugin.py 的目录
  → 跳过 plugin.disabled
  → 动态 import
  → 实例化插件
  → 读取 manifest / config schema
  → 注入 PluginContext
  → 绑定生命周期 handler
  → 注册工具
  → 注册 tool hook
  → 收集 phase modules
  → initialize
  → 成功则 loaded，失败则回滚
```

### 12.2 建议支持的插件能力

| 能力 | 入口 | 用途 |
|---|---|---|
| PhaseModule | `before_turn_modules()` 等方法 | 精确插入生命周期 |
| EventBus handler | `@on_before_turn` 等装饰器 | 观察或改写上下文 |
| Tool hook | `@on_tool_pre` | 拦截工具、拒绝高风险操作 |
| Tool registration | `@tool` | 给模型增加新工具 |
| Channel | `channels()` | 增加新消息通道 |
| Proactive module | `proactive_modules()` | 参与主动推送模块链 |
| Dashboard panel | 插件静态资源与 API | 增加插件面板 |

### 12.3 PhaseModule 与 slot

建议保留基于 `slot / requires / produces` 的模块编排机制，而不是只支持简单的 early/late 钩子。

原因：

- 插件插入点更精确。
- 内置模块和插件模块可以统一编排。
- 后续 phase 增多时不会失控。
- 更容易输出 `--inspect-modules` 调试树。

推荐保留的 slot 前缀：

| 前缀 | 作用 |
|---|---|
| `session:extra_hint:*` | BeforeTurn 追加 hint |
| `session:abort_reply` | BeforeTurn 直接终止本轮 |
| `reasoning:extra_hint:*` | BeforeReasoning 补充推理规则 |
| `reasoning:abort_reply` | BeforeReasoning 跳过 LLM |
| `prompt:section_top:*` | 注入 system prompt 顶部 |
| `prompt:section_bottom:*` | 注入 system prompt 底部 |
| `step:early_stop_reason` | 提前停止工具循环 |
| `step:telemetry:*` | 记录 step 级别指标 |
| `persist:assistant:*` | 写入 assistant 消息 extra |
| `outbound:media:*` | 附加媒体输出 |
| `turn:extra:*` | 写入 committed turn extra |

## 13. 工具系统规划

工具建议统一注册到 `ToolRegistry`，并且支持风险元信息。

### 13.1 工具分类建议

- `meta_common`：只读工具、网页读取、文件读取、搜索等。
- `memory`：回忆、记住、忘记、检索相似记忆。
- `schedule`：提醒、定时任务、未来待办。
- `mcp`：外部 server 工具发现与调用。
- `spawn`：子任务、子 Agent、后台委派。
- `channel_push`：主动消息投递。

### 13.2 工具元数据建议

每个工具建议保留以下属性：

- `risk`
- `always_on`
- `search_hint`
- `source_type`
- `source_name`

### 13.3 安全机制

必须从第一阶段就纳入：

- 工具执行前统一走 hook 链。
- 高风险工具必须可拒绝。
- Shell 工具必须限制交互式命令和危险命令。
- 工具循环必须有 guard，防止重复调用。
- Spawn 工具也必须继承同一套 hook。

## 14. 通道、消息总线与会话

### 14.1 通道规划

建议分阶段实现：

| 阶段 | 通道 |
|---|---|
| MVP | CLI、IPC、Telegram 三选二 |
| Beta | Telegram + CLI + IPC |
| 后续 | QQ、Feishu、插件通道 |

### 14.2 消息总线设计

建议保留两类总线：

- `MessageBus`：传输入站消息和出站消息。
- `EventBus`：广播生命周期事件、工具事件和观测事件。

### 14.3 会话与身份

`SessionManager` 至少要负责：

- 根据 channel/chat_id/user_id 获取或创建 session。
- 持久化消息和元数据。
- 维护最近活跃时间。
- 为主动推送提供 last_user_at、cooldown 等信号。

中长期要补齐：

- 不同通道 identity merge。
- message dedupe。
- attachment store。

## 21. 风险点与约束

| 风险 | 说明 | 建议 |
|---|---|---|
| LLM 输出不稳定 | 工具调用、JSON、structured output 可能失真 | 优先选工具调用稳定的模型，保留 retry 和 repair |
| prompt cache 失效 | 高频改 `MEMORY.md` 会降低缓存收益 | 使用 `PENDING.md` 缓冲并交给 optimizer 低频归档 |
| 主动推送打扰用户 | 阈值和 cooldown 配置不当会造成噪音 | 记录 tick log，优先保守阈值 |
| 工具副作用过大 | Shell、MCP、消息推送不可逆 | 风险标签 + hook + 审计日志 |
| 插件顺序复杂 | PhaseModule 依赖错误会导致行为异常 | 提供 `--inspect-modules` 和 slot 校验 |
| 多通道身份混乱 | 同一用户可能产生多个 session | 先单通道 MVP，后续补 identity merge |
| Dashboard 暴露风险 | 后台具备修改记忆和消息能力 | 默认内网、增加认证 |

## 23. 开发时必须遵守的实现准则

为了保证后续工程不会越做越乱，建议把以下规则当成硬约束：

- 所有核心能力都必须先抽象协议，再写具体实现。
- 所有后台任务都必须可关闭、可观测、可追踪。
- 所有工具都必须有风险标识。
- 所有会影响回复行为的扩展点都要能在日志或 Dashboard 中看见。

- 所有插件加载失败都必须回滚，不允许半加载状态。

## 24. 总结

`Memoria-Agent` 不应被实现成一个“调用一次 LLM 回答问题”的项目，而应被实现成一个长期运行的个人 Agent 系统。它的核心价值来自五条主线：

- 被动对话链路清晰：消息、会话、记忆、prompt、工具、回复、持久化分层明确。
- 主动推送链路完整：调度、数据源、判断、去重、投递和 Drift 能闭环。
- 记忆系统双层设计：Markdown 保障可解释性，向量层提供语义能力。
- 插件系统深入核心：生命周期、工具、hook、通道、Dashboard 都可扩展。
- Dashboard 支撑运维：会话、消息、记忆、主动推送、错误和 trace 都可观察。

后续开发建议优先围绕以下四个方向推进：

- 先完成最小闭环，再扩展能力。
- 先保证记忆质量和安全边界，再追求更多工具和通道。
- 先把启动装配、生命周期和插件抽象打稳，再增加复杂功能。
- 先让系统可观察、可调试、可回滚，再做高级自动化。
