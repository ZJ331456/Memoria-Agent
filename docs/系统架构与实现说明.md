# Memoria-Agent 系统架构与实现说明

## 1. 文档结论

参考 `akashic-agent` 的真实代码后，完整 Agent 系统可拆为 **8 个部分**：配置与启动、消息与会话、Agent 推理、长期记忆、工具与插件、主动触达、通道接入、Dashboard 与可观测性。

本仓库当前完成的是可独立运行和测试的核心 MVP：第 1、2、3、4、8 部分已经形成闭环；第 5、6、7 部分保留了清晰的演进边界，尚未用不可用的空实现占位。

```text
Browser / shadcn Dashboard
            │ HTTP JSON
            ▼
        FastAPI API
            │
      AgentService ────── LLMClient ────── OpenAI-compatible API
            │
        SQLite Store
       ┌────┴─────┐
   Sessions     Memories
```

## 2. 第一部分：配置与启动

### 2.1 参考项目做法

`akashic-agent` 由 `main.py` 解析命令，再由 `bootstrap/app.py` 和 `bootstrap/tools.py` 分成应用运行时与核心运行时两层装配。这样通道、Dashboard、后台任务的生命周期由应用层管理，而模型、会话、记忆和工具由核心层管理。

### 2.2 Memoria 实现

- `main.py`：解析 `--config`、`--host`、`--port` 并启动 Uvicorn。
- `memoria/config.py`：读取 TOML，解析 `${ENV_VAR}`，支持 `extends` 配置继承。
- `config.example.toml`：默认继承 `../akashic-agent/config.toml`。
- 配置优先级：命令行指定文件 > `MEMORIA_CONFIG` > `config.toml` > `config.example.toml`。

系统使用三类模型：

| 模型 | 作用 | 当前配置来源 |
|---|---|---|
| main | 最终对话生成 | `[llm.main]` |
| fast | 低成本记忆提取 | `[llm.fast]`，缺失时回退 main |
| embedding | 后续语义向量检索 | `[memory.embedding]` |

安全边界：`Settings.public_dict()` 只向 Dashboard 返回模型名、Base URL 和是否已配置，永不返回 API Key。

## 3. 第二部分：消息与会话

### 3.1 设计职责

会话层负责把不同来源的连续消息组织成上下文，也是记忆 consolidation、主动推送和审计回放的基础。参考项目中它由 `session/manager.py`、`session/store.py` 和消息总线共同承担。

### 3.2 当前数据模型

`sessions` 保存会话标题和时间；`messages` 保存角色、正文和所属会话。SQLite 开启 WAL 与外键，删除会话时级联删除消息。

关键约束：

- 会话首次发送消息后，标题自动取用户消息前 28 字。
- 发送前先持久化用户消息；模型失败时用户输入仍保留，便于重试和排障。
- 只向模型发送 `memory_window` 范围内的最近消息，避免上下文无限增长。
- 所有时间使用带时区的 UTC ISO 8601 字符串。

## 4. 第三部分：Agent 推理核心

### 4.1 参考项目链路

`akashic-agent` 的被动回复包含 BeforeTurn、BeforeReasoning、PromptRender、Reasoner、AfterReasoning、AfterTurn 等 Phase，并允许插件插入模块。完整实现适合扩展型运行时，但 MVP 首先需要保证核心数据流正确。

### 4.2 当前链路

```text
POST chat
  → 校验会话
  → 保存 user message
  → 搜索相关记忆，并用高权重记忆补足
  → 拼装 system prompt + memory context + recent messages
  → 调用 main model
  → 保存 assistant message
  → fast model 提取长期记忆
  → 返回回复与本轮新增记忆
```

`memoria/llm.py` 直接调用 OpenAI Chat Completions 兼容接口，因此 DeepSeek、DashScope 和其他兼容服务可以共用。超时为 120 秒，供应商错误转换为 API 的 502 响应，前端会展示具体错误。

当前未实现工具调用循环，这是下一阶段最重要的核心增强。届时应加入 `ToolRegistry`、结构化 tool calls、最大迭代数、取消令牌和副作用审计。

## 5. 第四部分：长期记忆

### 5.1 参考项目的双层记忆

`akashic-agent` 同时使用人类可读的 Markdown 层与 `memory2` 语义层。Markdown 包括 HISTORY、PENDING、RECENT_CONTEXT、MEMORY 等文件；向量层包含 query rewrite、HyDE、retriever、dedup、profile extractor 与 post-response worker。

### 5.2 当前 MVP 策略

当前 `memories` 表保存：正文、类型、重要度、来源和时间。记忆类型包括 fact、preference、profile、goal、procedure。

检索分两步：先用当前用户输入做 `LIKE` 精确文本召回；不足三条时用最高重要度的近期记忆补齐。该方案可解释、无需额外依赖，适合作为 MVP 基线，但不等于语义检索。

每轮回复后，快速模型按严格 JSON 约定最多提取三条记忆，并遵循以下规则：只记未来有帮助的用户信息，不记临时问题，不记凭据，不把助手内容当用户事实。解析或模型失败不会影响主回复。

下一阶段应接入配置中的 embedding 模型，新增向量表、余弦召回、语义去重和人工撤销记录；再加入 Markdown consolidation，保持“机器可检索 + 人类可编辑”。

## 6. 第五部分：工具与插件

参考项目的插件能通过 PhaseModule、EventBus、tool pre-hook 和工具注册介入运行时，并带有依赖排序和失败回滚。

Memoria 的建议分层：

- `ToolRegistry`：工具 schema、执行器、权限级别、超时和结果裁剪。
- `PhaseRunner`：按 slot 与依赖拓扑执行生命周期模块。
- `EventBus`：提供观察事件，不直接修改主链路状态。
- `PluginManager`：发现 manifest、加载、注册、失败回滚和禁用。
- `SafetyPolicy`：文件写入、Shell、外发消息等副作用执行前确认。

这一部分当前未实现，因为没有工具执行闭环时提前搭插件外壳只会增加错误抽象。推荐先完成 read-only 工具，再加可撤销写工具。

## 7. 第六部分：主动触达

参考项目的 `proactive_v2` 是独立循环：根据 presence/energy 动态决定轮询间隔，读取 alert、content、context 三路来源，再经 gate、judge、resolve、deliver；无内容时可以进入 Drift 执行 SKILL.md。

Memoria 后续实现需要具备：

- 独立于聊天请求的 asyncio 后台循环。
- 每个目标单独的冷却时间、静默时段和预算。
- 数据源游标与内容去重，避免重复推送。
- 判定 trace，能够说明为何发送或跳过。
- 发送与决策分离，发送必须经过通道权限检查。
- Drift 任务限定工作目录、步骤数和副作用权限。

主动触达当前不在 MVP 启用，防止未经用户设置目标和静默时间就产生外发行为。

## 8. 第七部分：通道接入

参考项目通过 MessageBus 隔离 Telegram、QQ、CLI、IPC 等通道。标准消息至少应包含 channel、chat_id、sender_id、content、attachments 和 trace_id。

当前 Web Dashboard 直接调用 HTTP API，是本地核心功能的测试通道。后续 Telegram/QQ 不应直接调用模型，而应统一发布 inbound message，再从 outbound queue 消费回复。这样记忆、工具、审计和取消逻辑只维护一份。

## 9. 第八部分：Dashboard 与可观测性

前端位于 `frontend/`，使用 React、TypeScript、Vite、Tailwind，以及 shadcn/ui 同款的 Radix Slot、class-variance-authority、`cn()` 和 `components/ui` 组织方式。当前包含：

- 对话实验室：新建/切换/删除会话、发送消息、加载态、错误反馈。
- 长期记忆：搜索、添加、删除、类型/重要度/来源展示。
- 运行状态：会话、消息、记忆计数，三类模型连接配置和配置来源。
- 响应式布局：桌面侧栏与窄屏纵向布局。

后端 API：

| 方法 | 路径 | 用途 |
|---|---|---|
| GET | `/api/health` | 存活检查 |
| GET | `/api/overview` | 数据规模与脱敏模型状态 |
| GET/POST | `/api/sessions` | 会话列表/创建 |
| GET | `/api/sessions/{id}/messages` | 消息历史 |
| POST | `/api/sessions/{id}/chat` | 执行完整对话与记忆提取 |
| DELETE | `/api/sessions/{id}` | 删除会话及消息 |
| GET/POST | `/api/memories` | 搜索/新增记忆 |
| PATCH/DELETE | `/api/memories/{id}` | 修改/删除记忆 |

生产化可观测还应增加 trace_id、模型延迟和 token、工具调用链、记忆召回依据、结构化错误表和健康指标。

## 11. 验收方法

1. 构建前端并启动后端，访问 2237。
2. 状态页确认 main/fast/embedding 都显示模型名称且密钥为“已配置”，页面不可看到密钥原文。
3. 新建对话，发送“记住我喜欢简洁的回答”，应获得模型回复。
4. 打开长期记忆，确认快速模型抽取的 preference；如果模型判断不值得记录，可手动添加。
5. 新开会话询问“我偏好怎样的回答”，检查记忆是否被检索并影响回复。
6. 重启服务，确认会话、消息和记忆仍存在。

## 12. 推荐演进顺序

P1：embedding 语义检索、去重与记忆编辑；P2：工具调用循环与只读工具；P3：生命周期和插件；P4：Markdown consolidation；P5：主动触达与 Drift；P6：Telegram/QQ 通道；P7：完整 trace、权限沙箱和评估集。

核心原则是每一阶段都维持可运行闭环，不为目录齐全而创建无法验证的抽象。
